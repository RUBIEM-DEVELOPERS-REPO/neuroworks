# Skill: Retry with a different approach

## Goal

The customer asked us to try again. The previous answer missed something — they're not asking for a polish, they're asking for a DIFFERENT angle. The new answer should not look like a tweak of the prior one.

## What "different" means (pick at least one)

- **Different structure.** If the prior was a numbered list, try a table. If it was a memo, try a script or dialogue. If it was prose, try a checklist.
- **Different angle.** Same task, but from a different stakeholder's view (user / engineer / finance / customer / legal).
- **Different scope.** Zoom out (the strategic context) or zoom in (the one most important detail).
- **Different first move.** If the prior led with the solution, lead with the problem. If it led with theory, lead with an example.
- **Different deliverable.** If they asked for a draft and got a long one, try the smallest possible deliverable that meets the need.
- **Different tools / sources.** If the prior was pure synthesis, pull a vault note or web source. If it was a web dump, anchor in personal context.

## Process

1. **Read the prior attempt explicitly.** Don't pretend it didn't happen. The customer's already seen it; restating it wastes their time.
2. **Diagnose the miss in one line.** "Prior answer was too abstract", "Prior answer skipped the user view", "Prior answer was too long for the slack context".
3. **Pick ONE of the "different" axes above.** Don't blend several — clarity is the point.
4. **Build the new answer from that axis.** Not from the prior answer's structure.
5. **Open with one acknowledgment line** ("Take 2 — let's try this from the engineering angle / as a one-pager / by leading with the user problem"). Then the new answer.

## Anti-patterns to avoid

- **Polishing the prior.** "Here's a tightened version" is what the customer rejected.
- **Adding more.** Length is rarely the missing thing. If the prior was a 1-pager, a 3-pager is usually worse, not better.
- **Apologising at length.** "Sorry, you were right, let me try again, I should have…". Get to the new answer.
- **Same structure with different wording.** The customer said "different approach" — the bones changed, not just the paint.
- **Hedging the new answer.** "This might also miss the mark, but…" — own the second take.

## Pitfalls

- **Asking for clarification instead of trying.** If the prior was wrong, the customer has signalled what they want by what they rejected. Make a confident second swing; ask for clarification if you ALSO miss.
- **Treating retry as failure.** Iteration is the work. The retry is part of the deliverable, not a recovery from a mistake.
- **Producing the prior with a "wait there's more" coda.** Tacking 2 paragraphs onto the prior answer is not a retry.
- **Ignoring the persona's lane.** If you're Casey on a customer email retry, the new angle still has to be a customer email, not a SQL query.
