# Skill: Memo writing

## Goal

A memo is what you write when an email is too short and a doc is too much. One page, one position. Everyone who reads it should walk away knowing the same thing.

## Format

```
TO: <audience — team / leadership / "all engineering">
FROM: <your name, role>
DATE: <YYYY-MM-DD>
RE: <subject — concrete, 8 words or fewer>

## Bottom line
<2-3 sentences. The thing you'd say if the room only had 30 seconds.>

## Background
<3-5 sentences of context. Why this came up. What changed.>

## What we're doing
<The decision / plan / position. Specific verbs, owners, dates.>

- <Action 1 — owner — date>
- <Action 2 — owner — date>

## What this means for you
<One paragraph per affected group. Concrete impact, not abstraction.>

## Open questions
<Things we haven't decided yet. Honest. Don't hide unknowns.>
```

## Rules

- **Lead with the conclusion.** "Bottom line" is paragraph 1, not paragraph 4.
- **Names + dates beat verbs alone.** "Will ship soon" is a wish; "Alex ships v2 by Nov 22" is a memo.
- **Cap at 1 page.** ~500 words. If you're past 700, you're writing a doc — link to it from a shorter memo.
- **No marketing voice.** Write how engineers talk in standups: declarative, specific, low adjectives.
- **One memo, one topic.** A "Q4 plan AND hiring update" memo gets read like both got skipped.

## Pitfalls

- Burying the decision in "background" → leadership stops reading.
- Listing risks without naming who's mitigating them → looks like a complaint, not a plan.
- Using passive voice for accountability ("will be addressed") → name the person.
- Sending without a "what this means for you" → readers ask back-channel "what does this mean for my team?" anyway.
