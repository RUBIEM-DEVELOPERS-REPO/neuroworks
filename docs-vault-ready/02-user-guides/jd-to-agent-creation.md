# Admin guide — Job description to agent

> Paste a real-world job description; NeuroWorks extracts the role,
> responsibilities, tone, and authority, then turns it into a working
> agent (a persona) on the Personas page in under a minute.

## TL;DR

1. Open `/personas`.
2. Click `+ New persona`.
3. Type a name (e.g. `Maya`), paste the JD into the Job description box.
4. Wait ~3-5 s for the live preview to populate `Role`, `Description`,
   `Tone`, and `Responsibilities`.
5. (Optional) Override anything in the preview that's wrong.
6. Click `Create`. Persona is saved + 6-8 starter templates are auto-
   generated under `/templates` for that persona.

## How extraction works

`extractPersonaMetadata(jd)` calls Ollama (local) to extract a JSON object
of `{ role, description, tone, responsibilities[] }` from the JD text. The
small fast model is used by default; if the JSON comes back malformed or
truncated, a regex heuristic fallback fills in the gaps so creation never
hard-fails.

Debounced live preview fires 800 ms after you stop typing, so the right
panel updates as you refine the JD.

## What gets created

For each new persona:

- **Persona record** at `data/personas.json` with `{ id, name, role,
  description, tone, responsibilities, jobDescription, createdAt }`.
- **Starter templates** under `/templates` → "Custom", tagged with the
  persona id, covering the persona's signature deliverables. The
  `persona-templates.ts` module knows the curated table for built-ins;
  custom personas get LLM-drafted starter templates.
- **System suffix** generated by `personaSystemSuffix(persona)`, prepended
  to every chat / team task the persona handles so the planner biases tool
  choice and output shape toward the role's conventions.

## The lane gate

Every persona except `clawbot` (the generalist) operates under a lane
discipline. When a task arrives that's clearly outside their role's
authority, `checkLaneFit(persona, text)` returns `inLane: false` and the
chat surfaces an inline refusal with a hand-off recommendation:

> *"This needs an Account Executive — I'm the Customer Success Manager.
> Would you like me to hand this to Drew?"*

This runs as a 1-2 s LLM check BEFORE the planner spins up, so a wrong-
hire task short-circuits instead of producing apologetic fake SQL.

## Activating and deactivating

- **Activate:** click `Activate` on the persona's card. The header on
  Chat now shows the persona's greeting.
- **Deactivate:** click `Deactivate` on the active persona, or activate
  `clawbot` (the generalist).
- **Auto-route fallback:** when no persona is active OR `clawbot` is on,
  the persona auto-router picks the right specialist per-task from a
  17-persona regex catalog. The auto-route only fires when one persona
  is a confident single winner; otherwise the generalist handles it.

## Templates lifecycle

Per-persona starter templates auto-refresh:

- **Backfill on server boot** — any persona missing templates (custom
  personas created before the curated table existed, or built-ins seeded
  before the refresher landed) gets their starter set generated.
- **Manual refresh** — `Refresh templates` button on each persona card
  re-runs the LLM-drafter against the persona's current `jobDescription` +
  `responsibilities`. Useful after editing the JD.
- **Custom templates** persist in `data/custom-templates.json` with an
  `mtime`-aware cache so out-of-band seeders (CLI scripts) reload without
  needing a server restart.

## Deleting a persona

`DELETE /api/personas/:id` removes the persona record AND all its tagged
starter templates. The response includes `removedTemplates` so you know
how many templates were cleaned up. Active persona reverts to none (so
the auto-router kicks in for the next chat).

## API reference (admin endpoints)

- `GET /api/personas` — list all + the active id
- `POST /api/personas` — create from `{ name, jobDescription, tone? }`
- `POST /api/personas/preview` — LLM-extracted metadata only, no save
- `POST /api/personas/:id/activate` — set active
- `POST /api/personas/deactivate` — clear active
- `DELETE /api/personas/:id` — remove + clean templates
- `POST /api/personas/:id/refresh-templates` — re-run starter drafter

All endpoints are loopback-bound (`127.0.0.1`) behind the
`originGuard` middleware — they aren't reachable from outside the
machine.

## Best practices

- **Use real JDs, not abstract role descriptions.** "Senior Customer
  Success Manager for B2B SaaS, owns NRR for accounts $50K-$500K ARR, leads
  weekly QBRs" beats "manages customer relationships" by miles.
- **Spell out the boundary in the JD.** If the role should refuse certain
  asks, write that into responsibilities ("Does NOT: write code, build
  models, draft contracts"). The lane gate picks it up.
- **Customise tone explicitly.** "Direct, plain English, no hedging" or
  "Warm, conversational, light on jargon" — the persona suffix carries
  the tone into every response.
